# InkMuse 产品规划书

## 1. 产品愿景

打造一个开源、可扩展、可自部署的 AI 小说创作平台，让创作者通过自然对话把“灵感”快速转化为“可持续连载的小说草稿”。

## 2. 产品定位

- 产品类型：对话式 AI 小说创作与迭代平台
- 核心目标：降低创作门槛、提升稳定出稿效率、保障长篇一致性
- 差异化：结构化创作资产管理 + 一致性守卫 + 对话/手动双微调

## 3. 目标用户

- 网文作者：追求稳定日更和剧情推进效率
- 编剧/策划：重视设定严谨性与剧情结构

## 4. 产品核心问题

- 灵感难落地：从想法到设定/大纲耗时长
- 长篇易崩：角色设定、时间线、剧情逻辑常冲突
- 迭代难管理：改稿过程缺乏清晰的变更管理与当前稿确认机制
- 成本不可见：模型调用成本、失败率难量化

## 5. 产品原则（对齐当前设想）

- 对话优先：核心资产都可通过 AI 对话创建与修改
- 轻量起步：项目初始仅创建书名与简介，先开题再深化
- 双轨微调：以对话为主，手动编辑仅做局部补充与确认
- 先闭环后扩展：V1 只做“灵感到单章并完成当前稿确认”最小可用闭环
- 单阶段单主形态：每阶段只保留一个主流程，避免并行复杂模式
- 技术约束：后端架构固定为 Hertz + Eino；前端技术栈暂不锁定，优先保证 API 与领域模型稳定

## 6. 核心功能规划

| 模块       | 功能                                             | 用户价值                     | 优先级 | 版本落点              |
| ---------- | ------------------------------------------------ | ---------------------------- | ------ | --------------------- |
| 项目入口   | 对话创建项目（书名、简介）                       | 最低门槛开题                 | P0     | V1                    |
| 设定工坊   | 世界观、角色、大纲生成与编辑                     | 快速形成创作骨架             | P0     | V1                    |
| 章节生成器 | 基于大纲生成单章、续写、局部重写                 | 稳定出稿                     | P0     | V1                    |
| 插件接口   | OpenAI 兼容中转站接口接入、Prompt 模板可配置调整 | 模型接入灵活、生成可控性提升 | P0     | V1                    |
| 一致性守卫 | 角色冲突、时间线冲突、剧情矛盾检测与修复建议     | 降低崩书风险                 | P1     | V1.1                  |
| 导出中心   | Markdown/TXT/EPUB 导出                           | 便于发布与协作               | P1     | V1.1                  |
| 成本看板   | token、调用次数、失败率、平均耗时可视化          | 可控运营成本                 | P1     | V1.1（V1 仅采集埋点） |

## 7. MVP 范围（V1，功能冻结）

- 对话创建项目（仅书名与简介）
- 世界观/角色/大纲生成与管理
- 资产双轨微调（对话修改 + 手动编辑）
- 单章生成、续写、局部重写
- 当前稿确认（章节状态切换为“已确认当前稿”）
- OpenAI 兼容中转站接口接入与 Prompt 模板可配置调整
- 后端基线采用 Hertz + Eino

## 8. V1 非目标（明确不做）

- 协作审稿
- 多模型复杂路由策略
- 创作资产版本系统（版本记录/对比/回滚）
- 一致性守卫（冲突检测与修复建议，延后到 V1.1）
- 批量章节生成与质量评分（延后到 V1.1）
- 导出中心（Markdown/TXT/EPUB，统一延后到 V1.1）
- 成本看板（可视化面板延后到 V1.1，V1 仅保留指标埋点）

## 9. 用户核心流程

1. 用户输入一句话灵感，与 AI 对话创建项目
2. 系统确认并保存：书名 + 简介
3. 用户继续通过对话生成世界观、角色、大纲
4. 用户以对话微调为主，必要时手动编辑补充关键细节
5. 系统基于当前大纲生成单章草稿（可续写/局部重写）
6. 用户确认“当前稿”（章节状态更新为已确认当前稿）并继续迭代
7. 保留在平台内持续连载迭代（V1 不包含导出）

## 10. 核心数据对象（建议）

- Project：书名、简介、状态、创建信息
- Asset：世界观/人物/大纲（结构化字段 + 自由文本）
- Chapter：章节草稿、修订稿、定稿状态、current_draft_id、current_draft_confirmed_at、current_draft_confirmed_by
- Conversation：对话上下文与指令历史、关联对象类型、关联对象 ID
- GenerationRecord：生成任务类型、输入快照引用、输出章节 ID、token 使用量、耗时、状态
- MetricEvent：事件名、项目 ID、章节 ID、事件时间、统计标签
- ConsistencyIssue（V1.1+）：冲突类型、严重级别、修复建议、处理状态

## 11. 产品路线图

### V1（0-3 个月）

- 目标：打通“灵感到单章并完成当前稿确认”的最小闭环
- 结果：用户可在 30 分钟内完成“开题-设定-大纲-首章草稿-当前稿确认”链路
- 架构基线：完成 Hertz + Eino 后端最小可用架构，前端保持技术选型开放
- 边界说明：不承诺导出、一致性守卫、批量生成、质量评分可用

### V1.1（3-5 个月）

- 新增：一致性守卫（冲突检测与修复建议）、批量章节生成、风格模板、质量评分、导出中心（Markdown/TXT/EPUB）、成本看板可视化
- 强化：一致性检测维度与修复建议质量、成本分析与模板效果归因

### V2（5-9 个月）

- 新增：协作审稿
- 增强：多模型路由与成本策略优化

## 12. 核心指标（KPI）

- 首次完整创作闭环完成率（分子：在统计窗口内完成“项目创建->首章生成->当前稿确认”的项目数；分母：窗口内创建项目总数；窗口：自然周；来源：Project/Chapter/MetricEvent）
- 30 分钟内首章产出成功率（分子：项目创建后 30 分钟内完成首章生成的项目数；分母：窗口内创建项目总数；窗口：自然周；来源：Project/GenerationRecord/MetricEvent）
- 单万字平均生成成本（分子：窗口内生成相关总成本；分母：窗口内生成总字数/10000；窗口：自然周；来源：GenerationRecord）
- 章节草稿采纳率（分子：标记为“已确认当前稿”的章节草稿数；分母：生成章节草稿总数；窗口：自然周；来源：Chapter/MetricEvent）
- Prompt 模板配置使用率（分子：使用自定义模板发起的生成任务数；分母：生成任务总数；窗口：自然周；来源：GenerationRecord）
- 一致性问题检出率与修复采纳率（V1.1+，来源：ConsistencyIssue）

## 13. 风险与应对

- 模型输出不稳定：触发阈值=单日结构化解析失败率 > 15%；Owner=后端负责人；SLA=24 小时内恢复到 <= 10%；应对=结构化输出约束 + 自动重试 + 结果校验 + 失败样本回放
- 长篇一致性下降：触发阈值=抽检章节中关键设定冲突率 > 20%；Owner=产品负责人；SLA=48 小时内完成规则加固并回归抽检；应对=V1 关键事实锁定 + 人工复核兜底，V1.1 上线一致性守卫
- 范围膨胀：触发阈值=单迭代新增需求中 V1 非目标占比 > 30%；Owner=产品经理；SLA=当前迭代内完成范围回退与需求重排；应对=V1 功能冻结，新增需求进入下一版本池
- 维护成本上升：触发阈值=连续两周交付延期或关键模块变更失败率 > 20%；Owner=技术负责人；SLA=一周内完成模块边界审查与改造计划；应对=模块化边界治理 + 覆盖率门槛 + 复杂度巡检

## 14. 成功定义

- 用户可从 0 到 1 完成可读小说草稿
- 长篇迭代过程中主要设定冲突显著下降
- 用户能在对话与手动微调间自由切换并保持创作节奏
- 开发者可在低成本下扩展模型与功能
- 产品可持续演进，不因架构复杂度失控而停滞
